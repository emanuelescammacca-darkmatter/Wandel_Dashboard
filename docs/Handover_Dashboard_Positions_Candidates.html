About this document

The UI was built design-first: every screen is final in look, feel and interaction, but reads from constants inside the React components. Your job is to replace those constants with real data without changing the UI. For every block this document tells you:

  • Where the value lives today — file + constant name, so you can find and delete the mock.
  • Where it must come from — database field, computed rule, recruiter input, Sophia call, or LLM.
  • The exact shape — the TypeScript type the component consumes is the API contract.
  • The behavior — interactions and cross-page side-effects you must preserve.
Source tags used throughout: Mock hard-coded today, must be replaced Database stored field Computed deterministic backend rule Sophia Call structured output of the screening call LLM generated by a model (prompt in §08) Manual recruiter input in the UI
Property tags: Editable user can change it in this view Read-only display only
Ground rule: everything shown in the UI must be standardizable — either a stored field, a deterministic rule over stored fields, or a clearly-labeled LLM output produced by a fixed prompt per call/candidate. No free-form, manually-maintained display strings.
Out of scope: the internal recruiter area (sidebar “Candidates” → candidate detail, “Positions”, “Clients”) is not part of this handover — do not refactor or translate it as part of this work. This document covers exactly three client-facing areas: Dashboard, Position Workspace, Candidate Profile.

Start here — five things to internalize before reading on

  1. Everything you see is mock. The UI is finished; not a single value comes from a server. Appendix A.3 lists every mock constant and the file it lives in.
  2. The central record is the application (candidate × position) — not the candidate. Stage, match score, criteria, notes and assessment all belong to an application. One person applying to two positions = two applications with independent stages. (§07)
  3. One stage enum, four surfaces. The same pipeline stage is shown/edited on the dashboard board, the position table, the position board and the profile dropdown. The prototype already syncs them through a shared store (src/lib/talentStore.ts) — replace that store with the API, keep the behavior. (§03)
  4. Intelligence is offline. All LLM outputs (match score, assessment, criteria, questions) are generated once per event — after a call or at position creation — and stored. Nothing is generated at page load. (§08)
  5. Colors are rules, not data. Every traffic light in the UI is a deterministic threshold over a stored value. The complete rule list is §09 — implement it once, centrally.

Architecture & data flow

Five upstream sources feed the database; the three feature areas read from it. The Sophia voice call is the most important source — most decision-relevant fields are structured answers from that call.

   screening questions            │     └──────────────────┘  ◀───

Key consequence of this architecture: the stage of an application is written once and read everywhere (dashboard board, position table, position kanban, comparison, profile header, profile evaluate-dropdown). See §03.

Stalled / nudge logic

Computed A card is stalled when stage is not terminal AND days_in_stage ≥ 5 (STALL_DAYS = 5 in Dashboard.tsx). The board header shows “N need a nudge”; stalled cards get an amber “{days}d · nudge” chip. days_in_stage = floor((now − stage_entered_at) / 86400s).

Why applications matters: stage, match score, criteria, must-haves, notes and assessment are all per position, not per person. The prototype blurs this (one global mockCandidates); the schema above fixes it. A candidate appearing in two positions has two application rows with independent stages and scores.

Deployment — embedding the tasks in AWS

Reuse the pattern from the existing call-analyser: AWS Bedrock for inference, event-driven and serverless around it. Nothing here is realtime. 

 EVENTS                          ORCHESTRATION                      INFERENCE + STORAGE
 ─────────────────────           ───────────────────────────        ─────────────────────────
 ElevenLabs webhook        ──▶   EventBridge bus                    Bedrock (Claude, JSON out)
   call.completed                  │                                     │
 Backend events                    ├─ Step Function "post-call"  ──▶  P2+P3+P4 as ONE call
   position.created                │    validate JSON → write DB         │
   application.rejected            ├─ Lambda "position-ingest"   ──▶  P1 then P5 → pending review
                                   ├─ Lambda "drop-reason"       ──▶  P6 (skip if recruiter chose)
 EventBridge Scheduler     ──▶     └─ Lambda "weekly-insight"    ──▶  P7 → cache in weekly_metrics
   weekly cron
                                   All writes → RDS Postgres (§07)
                                   Prompt I/O logged to S3 (evals, debugging)
  • P2/P3/P4 are one Bedrock call — same input, one combined JSON schema. Step Functions handles retries (one retry on schema-validation failure, then dead-letter + alert).
  • P5 output is “pending review” — the recruiter confirms/edits the questions in the UI before the position goes live.
  • Prompts live in the repo, versioned; every stored output carries model + prompt_version + generated_at so results can be regenerated after prompt changes.
  • No realtime constraints — no streaming, no provisioned throughput. On-demand Bedrock is sufficient.

What is missing / open decisions

Not implemented in the prototype (backend must add)

  • Persistence everywhere — stage moves, notes, question edits are all local React state.
  • Position data beyond p1 — routes are parameterized (:positionId), but JD/screening/comparison content is only seeded for p1; other positions show empty states until the JD-ingestion pipeline (P1/P5) exists.
  • Auth / client scoping — the dashboard greets “Techbridge GmbH” statically; every query in §07 is implicitly WHERE client_id = ….
  • Stage persistence — the four surfaces already sync via the in-memory store (§03), but a page reload resets everything; stage writes must go to the API.
  • Interview scheduling — “Interview Mo 10:00” tags (4.2.2) imply an interviews table that doesn't exist anywhere yet.
  • Real notes — author, timestamps, list of past notes (6.2.3).
  • Event pipeline — weekly metrics (4.1, 4.3.1) need application_events + a weekly rollup job.
  • Geocoding — city → lat/lng at ingest for map + distance (R8).

Decided (10 Jun 2026)

Resolved — implement as specified

  • Hero metrics (4.1): the three most important categories from funnel + kanban — New Applications (funnel volume, weekly + delta) · Matched Candidates (funnel outcome) · Interviewing (kanban stage count, live).
  • Funnel stage mapping (4.2.1): Sourcing = COUNT applications via Meta lead ads · Screening = COUNT completed Sophia calls · Matched = COUNT matched to the position. The kanban pipeline is independent.
  • Comparison candidate set (5.2): all matched candidates of the position, including rejected ones — the matrix documents why someone is out.
  • Match score: deferred — not relevant for the first iteration. Treat the P2 caps (49/84) as placeholders; ship the score plumbing, revisit the model later.
  • Outreach funnel (4.5): purely decorative, independent of real data. No backend work — the channel counts stay static.
  • Dashboard secondary cards: legacy PULSE_BY_TITLE / plain PositionCard variants were removed from the codebase; every position renders the focus card.

Known prototype quirks (do not replicate)

  • The shared store lives in browser memory — a full page reload resets all stage moves to the seed values in talentStore.ts.
  • The “Saved ✓” note button fakes success.
  • Avatar colors are stored per candidate in talentStore.ts — generate deterministically from the id hash instead.
  • Aggregate numbers are scaled to the 8-candidate demo pool and internally consistent (funnel “Matched” = pool per position, weekly activity “all” = sum of the per-position trends, hero growth = sum of this week's sourcing deltas). Acquisition-side counts (Sourcing, contacted) are intentionally larger than the matched pool.
